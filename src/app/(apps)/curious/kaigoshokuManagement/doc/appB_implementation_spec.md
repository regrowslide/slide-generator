## 1. システム概要

- **名称**: 介護食管理システム（仮称）
- **目的**: 複数の介護施設からの日々の食事注文（食数）を取り込み、献立マスタに基づき必要な「製造総重量」「梱包内訳」「原材料発注量」を自動計算・出力する。
- **プラットフォーム**: Webアプリケーション（アプリAと同サーバー・同DBで稼働）

---

## 2. データモデル

### 2.1. 献立データの階層構造

献立データは以下の5階層で管理する。

```
┌─────────────────────────────────────────────────────┐
│ KgDailyMenu (日付)                                  │
│   └─ KgMealSlot (Category: 朝食、昼食、昼間食、夕食)│
│       └─ KgMenuRecipe (Menu: 献立)                  │
│           └─ KgMenuRecipe (Dish: 料理)              │
│               └─ KgRecipeIngredient (材料)          │
└─────────────────────────────────────────────────────┘
```

- **Menu（献立）**: `parentRecipeId = null` の `KgMenuRecipe`
- **Dish（料理）**: `parentRecipeId = Menu.id` の `KgMenuRecipe`

### 2.2. 主要モデル

#### KgDailyMenu（日付献立）
| フィールド | 型 | 説明 |
|-----------|-----|-----|
| menuDate | Date | 献立日付（ユニーク） |
| totalVegetableG | Float? | 1日合計野菜量(g) |
| pfc | String? | P:F:C比率（例: "15:32:50"） |

#### KgMealSlot（食事区分）
| フィールド | 型 | 説明 |
|-----------|-----|-----|
| dailyMenuId | Int | 日付献立への参照 |
| mealType | String | breakfast, lunch, snack, dinner |
| mealTypeName | String | 朝食, 昼食, 昼間食, 夕食 |
| total* | Float? | 栄養素合計（Energy, Protein, Fat等） |

#### KgMenuRecipe（献立レシピ）
| フィールド | 型 | 説明 |
|-----------|-----|-----|
| mealSlotId | Int | 食事区分への参照 |
| parentRecipeId | Int? | 親レシピID（Menu=null, Dish=MenuのID） |
| code | String | レシピコード（CSV取込元コード） |
| name | String | レシピ名 |
| category | String? | 主菜, 副菜, 主食, 汁物, デザート |
| unitWeight | Float? | 1人前の重量 |
| total* | Float? | 栄養素合計（計算値） |

#### KgRecipeIngredient（レシピ食材）
| フィールド | 型 | 説明 |
|-----------|-----|-----|
| menuRecipeId | Int | 料理(Dish)への参照 |
| ingredientMasterId | Int? | RcIngredientMasterへの参照（リンク済み） |
| ingredientCode | String | 食品標準成分表コード |
| ingredientName | String | 食材名 |
| amountPerServing | Float | 1人前の使用量(g) |
| *（栄養素） | Float? | 計算された栄養素値 |

---

## 3. 業務フロー概要

1. **マスタ準備**: 月次で「献立データ」を取り込む（外部データ連携）。
2. **注文取込**: 施設ごとの注文（CSVまたは画像）を取り込む。
3. **自動計算**: 「注文数 × 献立レシピ」で製造量を算出。「製造量 ÷ 歩留まり」で発注量を算出。
4. **出力**: 製造指示書、梱包指示書、発注書を出力。

## 4. 機能要件詳細

### 4.1. 献立CSVデータ取込機能

月次で外部の献立作成会社から提供される献立データ（CSV形式）を取り込む。

#### 4.1.1. 献立CSVのフォーマット

**ヘッダー部分:**
| 行 | 内容 |
|----|------|
| 2 | 「1日分献立表」「国の標準栄養価一覧表のコード」 |
| 3 | 日付（例: 2026年1月31日） |
| 4 | 事業者名称 |
| 5 | 空行 |
| 6 | カラムヘッダー（1日分献立、献立、レシピ、食品、...） |

**データ列の構造:**

| 列 | 内容 | 備考 |
|----|------|------|
| B列 | カテゴリ（朝食、昼食、昼間食、夕食） | 次のカテゴリになるまで空欄（=同上扱い） |
| C列 | 献立コード + 献立名 | 例: `11929 スクランブルエッグ【朝】` |
| D列 | 料理コード + 料理名 | 次の料理になるまで空欄（=同上扱い）。数値のみの場合は空欄とみなす |
| E列 | 追加情報 | （予備列） |
| F列 | 材料コード + 材料名 | 例: `12004 【12345】鶏卵 全卵 生`。「レシピ合計」行は料理の境界を示す |
| G列 | 1人分可食量（g） | この値をDBの `amountPerServing` に保存 |
| H列 | エネルギー（kcal） | |
| I列 | たんぱく質（g） | |
| J列以降 | 脂質、炭水化物、食塩相当量など | |

#### 4.1.2. パース処理ルール

1. **カテゴリ判定（B列）**:
   - 「朝食」「昼食」「昼間食」「夕食」等のキーワードで判定
   - 空欄の場合は直前のカテゴリを引き継ぐ

2. **献立判定（C列）**:
   - 空でない場合、新しい献立（Menu）を作成
   - `parentRecipeId = null` で `KgMenuRecipe` を登録

3. **料理判定（D列）**:
   - 空でない場合、新しい料理（Dish）を作成
   - D列に数値のみがある場合は空欄として扱う（材料行の可能性）
   - 親の献立を `parentRecipeId` に設定して `KgMenuRecipe` を登録

4. **材料判定（F列）**:
   - 「レシピ合計」の文字列は料理の合計行であり、次の料理への境界を示す → スキップ
   - それ以外は `KgRecipeIngredient` として登録

5. **重量保存（G列）**:
   - 材料の重量（g）をDBの `amountPerServing` に保存
   - これらの合計値がその料理の総重量となる

#### 4.1.3. コード体系

- **献立コード**: 5桁数値（例: 11929）
- **料理コード**: 5桁数値（例: 43823）
- **材料コード**: 食品標準成分表コード（例: 12004）
  - 【】内は追加情報（例: 【12345】）

### 4.2. 注文データ入力・取込機能

注文は施設ごとに「日付」と「食事区分（朝・昼・夕）」ごとの食数で管理される（料理名ごとの発注ではない）。

- **CSV取込（優先機能）**:
  - 既存の注文管理システムから出力されるCSVデータをインポートする機能を実装する。
  - **処理**: CSV内の「日付」「食事区分（朝/昼/夕）」「食数」「食形態（常食/刻み/ミキサー等）」を読み取る。
  - **運用**: 取引先（施設）を指定してからCSVを取り込むことで、施設ごとの注文として処理する。
- **画像・PDF取込（AI解析）**:
  - FAXやメール（PDF/画像）で届く注文書をAI（OCR）解析してデータ化する。
  - **信頼度スコアと確認フロー**:
    - AI解析時に「信頼度スコア」を算出し、スコアが低い場合はアラートを表示する。
    - 元画像と解析結果を並べて表示し、ユーザーが手動で修正・確定するステップを必須とする。
    - 紐付けができなかった項目は手動でマスタと紐付ける。

### 4.3. マスタ・データ管理機能

アプリA（原価計算アプリ）と一部マスタを共有し、効率化を図る。

- **献立データ取込（月次更新）**:
  - 外部の献立作成会社が作成したデータ（CSV/Excel）を取り込む。
  - **データ構造**: 「日付」「食事区分（朝/昼/夕）」「料理名」「使用食材」「1人あたり重量」が含まれる。
  - **重要要件（食材紐付け）**:
    - 現状の「標準化コード（栄養価ベース）」ではなく、アプリAと共通の「社内材料コード（発注ベース）」での紐付けが必要。
    - _対応策_: 献立作成会社側のCSVに「社内材料コード」を付与してもらうよう依頼中。これが実現すれば完全自動連携が可能となる。
- **原材料マスタ（アプリAと共有）**:
  - アプリAで管理する「食材名」「仕入れ先」「キロ単価」「歩留まり率」を参照する。
- **取引先マスタ**:
  - 施設名、配送区分などを管理。手動での追加・編集が可能。

### 4.4. 計算・展開ロジック（Core Logic）

「食数」から「材料」への変換および「歩留まり」の考慮を行う。

1. **料理展開（Explosion）**:
   - 入力された「朝・昼・夕の食数」を、その日の献立データと照合し、「料理ごとの製造必要数」に変換する。
   - 例：1月1日 朝食70食 $\rightarrow$ スクランブルエッグ70人前、ブロッコリーサラダ70人前。
2. **総重量計算**:
   - `1人あたり重量（献立データ） × 食数` で、料理ごとの必要総重量（kg）を算出する。
3. **原材料計算と歩留まり適用**:
   - 料理を構成する「原材料」に分解し、アプリAの「歩留まり率」を適用して発注必要量を計算する。
   - **計算式**: `必要重量 = (1人あたり使用量 × 食数) ÷ 歩留まり率`。
   - _例_: ほうれん草のお浸しを作る場合、茹でて40gになるなら、生のほうれん草は80g必要（歩留まり50%）といった計算を行う。

### 4.5. 出力・帳票作成機能

計算結果に基づき、以下の3種類の指示書・リストを出力する。

1. **製造指示書（現場向け）**:
   - **単位**: 日付ごと、料理ごと。
   - **表示内容**: その日に製造すべき料理名と「製造総重量（kg）」を表示する。
   - 施設ごとの内訳ではなく、まとめて製造するための合算値を表示する。
2. **梱包・配送指示書（仕分け向け）**:
   - **単位**: 日付ごと、取引先（施設）ごと。
   - **表示内容**:
     - 配送先施設名
     - 料理名
     - **食形態**: 「常食」「刻み」「ミキサー」などの区分。
     - **数量**: 「重量（kg）」または「パック数/人前数」を表示（発注数と一致させる）。
3. **原材料発注書（仕入れ向け）**:
   - **単位**: 仕入れ先ごと。
   - **表示内容**: 献立全体で必要な原材料を集計し、発注先ごとにPDF等で出力する。

## 5. 非機能要件・システム構成

### 5.1. インフラ・データベース

- **サーバー構成**: アプリAと同一サーバー・同一データベースを使用し、ランニングコストを抑制する（月額20ドル程度を想定）。
- **データ共有**: 「原材料マスタ」「仕入れ先マスタ」はアプリAと共通テーブルを参照する。

### 5.2. ユーザー権限・セキュリティ

- **権限区分**:
  - **システム管理者**: マスタ編集（献立取込、原材料編集）、ユーザー管理が可能。
  - **一般ユーザー**: 注文入力、計算実行、帳票出力のみ可能（マスタ操作不可）。
- **排他制御**: 複数ユーザーによる同時利用（別々の施設のデータ入力など）を想定した設計とする。

### 5.3. UI/UX方針

- **確認プロセスの重視**: AI解析や自動計算の結果をユーザーが確認・修正できるUIを徹底する（特に注文取込時）。
- **プロトタイプ検証**: 仕様が複雑であるため、開発初期段階でデモ画面（プロトタイプ）を作成し、認識齟齬がないか確認しながら進める。
