## 🔄 各フェーズの詳細

### セットアップフェーズ

#### 1. クライアント作成

**目的**: 分析対象となる組織を定義

**手順**:

1. `/hakobun/master/clients` にアクセス
2. 「新規クライアント」ボタンをクリック
3. クライアントIDと名前を入力
4. 保存

**入力項目**:

- `client_id`: 半角英数字とアンダースコア（例: `cafe_sample_01`）
- `name`: 表示名（例: `サンプルカフェ`）

**データベース**: `HakobunClient` テーブルに保存

#### 2. カテゴリマスター登録

**目的**: AIが使用するカテゴリの辞書を構築

**手順**:

1. `/hakobun/master/categories` にアクセス
2. クライアントを選択
3. 「新規カテゴリ追加」ボタンをクリック
4. カテゴリ情報を入力
5. 保存

**入力項目**:

- `category_code`: 一意の識別子（例: `cat_01`）
- `general_category`: 大カテゴリ（例: `店内`）
- `specific_category`: 小カテゴリ（10〜16文字推奨、例: `オシャレ・雰囲気が良い`）
- `description`: 説明（任意）

**カテゴリ名のルール**:

- **長さ**: 13文字前後（±3文字、10〜16文字）
- **方向性**: 「何がどう良い／悪い／要望か」が一読で分かる
- **抽象度**: 1段階抽象化（抽象的すぎず、細かすぎない）
- **具体例**:
  - ✅ `特定のパンへの美味しさ評価`
  - ✅ `接客や対応が雑・私語が多い`
  - ❌ `パンが美味しい`（抽象的）
  - ❌ `満足`（方向性がない）

**データベース**: `HakobunCategory` テーブルに保存

#### 3. ルール設定（任意）

**目的**: AIの判断基準を明示的に定義

**手順**:

1. `/hakobun/master/rules` にアクセス
2. クライアントを選択
3. 「新規ルール追加」ボタンをクリック
4. ルール情報を入力
5. 保存

**入力項目**:

- `target_category`: 対象カテゴリ（例: `備品・設備`）
- `rule_description`: ルール説明（例: `「音響」「BGM」「空調」に関する記述は「店内（雰囲気）」ではなく「備品・設備」に分類すること`）
- `priority`: 優先度（High / Medium / Low）

**データベース**: `HakobunRule` テーブルに保存

### 分析実行フェーズ

#### Phase 1: コンテキスト取得 (Retrieval)

**目的**: AIの判断に必要な情報をデータベースから取得

**処理内容**:

1. **カテゴリマスター取得**

   ```typescript
   const categories = await prisma.hakobunCategory.findMany({
     where: {hakobunClientId: client.id},
     orderBy: {categoryCode: 'asc'},
   })
   ```

2. **直近修正事例50件取得**

   ```typescript
   const corrections = await prisma.hakobunCorrection.findMany({
     where: {
       hakobunClientId: client.id,
       archived: false,
     },
     orderBy: {createdAt: 'desc'},
     take: 50,
   })
   ```

3. **全ルール取得**
   ```typescript
   const rules = await prisma.hakobunRule.findMany({
     where: {hakobunClientId: client.id},
     orderBy: [{priority: 'asc'}, {createdAt: 'desc'}],
   })
   ```

**取得データの用途**:

- **カテゴリマスター**: AIが使用可能なカテゴリのリスト
- **修正事例**: Few-Shot学習のための具体例
- **ルール**: AIの判断基準

#### Phase 2: 動的プロンプト構築 (Prompt Engineering)

**目的**: 取得した情報をSystem Promptに組み込む

**プロンプト構成**:

1. **役割定義**

   ```
   あなたは、投稿内容を「トピック単位」で整理してカテゴリ分類するアシスタントです。
   ```

2. **トピック分割ルール**
   - まずテーマを判定
   - 同一テーマは必ず統合（最優先ルール）
   - 異なるテーマが含まれている場合のみ分割
   - トピック単位は必ず原文そのまま
   - 文の途中で切らない（意味が完結する形で分割）

3. **カテゴリ名ルール**
   - 13文字前後（±3文字）
   - 方向性を含める
   - 1段階抽象化

4. **マスタカテゴリ一覧**

   ```
   - オシャレ・雰囲気が良い（内装、照明、空気感への好意的な評価）
   - BGMが大きすぎる（音量に関する不満）
   ...
   ```

5. **Few-Shot事例**

   ```
   入力「とにかく音楽がでかい！」→ カテゴリ: BGMが大きすぎる / 感情: 不満
   入力「抹茶チーズケーキ絶品です」→ カテゴリ: 特定のパンへの美味しさ評価 / 感情: 好意的
   ...
   ```

6. **追加ルール**

   ```
   - [High] 備品・設備: 「音響」「BGM」「空調」に関する記述は「店内（雰囲気）」ではなく「備品・設備」に分類すること
   ...
   ```

7. **出力形式**
   - JSON配列形式
   - 各要素に `sentence`, `category`, `sentiment`, `posi_nega`, `magnitude` を含める

**実装**: `src/app/api/hakobun/analyze/route.ts` の `buildSystemPrompt` 関数

#### Phase 3: AI推論 (Inference)

**目的**: Gemini APIを呼び出して分析結果を取得

**処理フロー**:

1. **API呼び出し**

   ```typescript
   const response = await fetch(`${GEMINI_API_ENDPOINT}?key=${apiKey}`, {
     method: 'POST',
     headers: {'Content-Type': 'application/json'},
     body: JSON.stringify({
       contents: [
         {
           parts: [{text: `${systemPrompt}\n\n【分析対象テキスト】\n${raw_text}`}],
         },
       ],
       generationConfig: {
         temperature: 0.2,
         topK: 40,
         topP: 0.95,
         maxOutputTokens: 4096,
       },
     }),
   })
   ```

2. **JSON解析**
   - レスポンスからJSON部分を抽出
   - 配列形式のJSONをパース
   - 型チェックとバリデーション

3. **結果保存**
   ```typescript
   await prisma.hakobunVoice.create({
     data: {
       voiceId: generatedVoiceId,
       rawText: raw_text,
       processedAt: new Date(),
       resultJson: parsedResult,
       hakobunClientId: client.id,
     },
   })
   ```

**出力形式**:

```json
[
  {
    "sentence": "かっぴーさんはじめお店の方皆さんが作り出されてる空気が素敵でめっちゃファンなのでこれからも応援しています！ありがとうございます！",
    "category": "気持ちの良い接客",
    "sentiment": "好意的",
    "posi_nega": "positive",
    "magnitude": 85
  }
]
```

### 結果確認・フィードバックフェーズ

#### 1. 分析結果の確認

**画面**: `/hakobun` の「テキスト分析」タブ

**表示内容**:

- トピック単位ごとの分析結果
- カテゴリ、感情、熱量スコア
- ポジネガ判定
- 統計情報（感情別の集計、平均熱量）

#### 2. 誤りの修正

**画面**: `/hakobun` の「フィードバック編集」タブ

**修正可能項目**:

- **カテゴリ名**: テキスト入力（10〜16文字推奨）
- **感情**: 好意的 / 不満 / リクエスト から選択
- **コメント**: 修正理由（任意）

**修正の流れ**:

1. 誤っている項目を特定
2. カテゴリ名または感情を修正
3. 修正理由をコメントに記入（任意）
4. 「フィードバック送信」ボタンをクリック

#### 3. フィードバック送信

**API**: `POST /api/hakobun/feedback`

**リクエスト例**:

```json
{
  "client_id": "cafe_sample_01",
  "voice_id": "voice_id_from_analysis",
  "corrections": [
    {
      "extract_index": 0,
      "original_sentence": "とにかく音楽がでかい！会話もしにくい。",
      "correct_category": "BGMが大きすぎる",
      "correct_sentiment": "不満",
      "reviewer_comment": "音量に関する不満なので設備カテゴリに分類"
    }
  ]
}
```

**処理**:

- 修正データを `HakobunCorrection` テーブルに保存
- `archived: false` として保存（次回の分析に含まれる）

#### 4. 修正データの蓄積

**データベース**: `HakobunCorrection` テーブル

**保存内容**:

- `rawSegment`: 元のトピック（原文）
- `correctCategoryCode`: 正解カテゴリコード
- `sentiment`: 正解感情
- `reviewerComment`: 修正理由（任意）
- `archived`: アーカイブフラグ（false = 有効）

### 次回分析への反映

**自動反映**:

- 次回の分析実行時、Phase 1で直近50件の修正事例を取得
- Phase 2でプロンプトにFew-Shot事例として組み込まれる
- AIが同じパターンを学習し、精度が向上

**反映タイミング**:

- 即座に反映（次回の分析から有効）

**将来の拡張**:

- ルール圧縮バッチ処理（週次）
  - 修正事例から共通パターンを抽出
  - 普遍的なルールとして `HakobunRule` テーブルに保存
  - ルール化済みの修正事例をアーカイブ

## 📈 精度向上のメカニズム

### 短期記憶（Few-Shot学習）

- **方法**: 直近50件の修正事例をプロンプトに含める
- **効果**: 即座に同じパターンの誤りを防ぐ
- **適用範囲**: 類似の言い回し、同じカテゴリの誤分類

### 長期記憶（ルール圧縮）

- **方法**: 修正事例から共通パターンを抽出してルール化
- **効果**: トークン数を節約しつつ、普遍的な判断基準を確立
- **適用範囲**: 頻繁に発生する誤分類パターン

## 🎯 ベストプラクティス

### カテゴリマスターの設計

1. **段階的な追加**: 最初は主要なカテゴリのみ登録し、運用しながら追加
2. **命名規則の統一**: カテゴリ名の長さと方向性を統一
3. **定期的な見直し**: 使用頻度の低いカテゴリを統合・削除

### ルールの設定

1. **優先度の適切な設定**: 重要なルールは High に設定
2. **具体性**: 抽象的なルールではなく、具体的な判断基準を記述
3. **定期的な更新**: 新しい修正事例に基づいてルールを更新

### フィードバックの送信

1. **即座の送信**: 誤りを発見したらすぐにフィードバックを送信
2. **コメントの記入**: 修正理由を記入することで、将来のルール化に役立つ
3. **一貫性**: 同じパターンの誤りは一貫して修正

## 🚨 トラブルシューティング

### 分析結果が期待と異なる場合

1. **カテゴリマスターを確認**: 該当するカテゴリが登録されているか
2. **ルールを確認**: 関連するルールが設定されているか
3. **修正事例を確認**: 類似の修正事例が蓄積されているか
4. **フィードバックを送信**: 誤りを修正してフィードバックを送信

### カテゴリが適切に分類されない場合

1. **カテゴリ名の確認**: 10〜16文字で方向性が明確か
2. **ルールの追加**: 判断基準をルールとして追加
3. **修正事例の蓄積**: 複数の修正事例を蓄積してFew-Shot学習を強化

### プロンプトが長すぎる場合

1. **修正事例のアーカイブ**: 古い修正事例をアーカイブ
2. **ルール化**: 頻繁に発生するパターンをルール化
3. **カテゴリの統合**: 類似カテゴリを統合して数を減らす
