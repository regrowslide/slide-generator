# 生成AI活用型「顧客の声」構造化APIシステム詳細設計書

## 1. システムコンセプト

**「新人の料理人（AI）に、秘伝のレシピ帳（DB）を持たせ、厨房（API）で働かせる」**
本システムは、不定形かつ大量（長文・短文混在）の顧客フィードバックを、組織固有の分析軸（カテゴリ・感性）に合わせて構造化・データ化するAPIである。

- **Input:** クライアントからの生テキスト（例：スポーツ観戦の感想、飲食店のアンケート）。
- **Process:** 文脈の分節化、ハイブリッドRAGによる動的プロンプト生成、推論。
- **Output:** 分析・タグ付けされた構造化JSONデータ。
- **Growth:** 人間による修正を「個別事例」として蓄積し、定期的に「ルール」へ圧縮してAIの精度を向上させる。

---

## 2. インプットデータ仕様

APIが受け取るリクエストの仕様です。ソースにある「スポーツ観戦の長文」や「飲食店の短文」など、あらゆる長さを許容します。

### 2.1 リクエストパラメータ

```
{
  "voice_id": "unique_transaction_id_001",
  "client_id": "cafe_sample_01",
  "timestamp": "2025-05-15T10:00:00Z",
  "raw_text": "ランチとは雰囲気が違うしよかったんですが、とにかく音楽がでかい！あと、ハンバーグが昼とあんまり見栄えに変化がなくて…その分値段の差がちょっとキツいなと感じました。"
}

```

- **raw_text**: 1件の「お客様の声」。数文字〜数万文字（スポーツチームの事例）まで対応。

## 3. データベース設計（Knowledge Base）

AIにコンテキスト（判断基準）を提供するための3つのテーブルを定義します。

### A. カテゴリマスター (Category Master)

組織として定義済みの「正解カテゴリ」リスト。AIのハルシネーション（勝手なカテゴリ作成）を防ぐための辞書。

- ソースの`generalCategory`と`category`に対応。

| category_id | general_category | specific_category      | description                        |
| ----------- | ---------------- | ---------------------- | ---------------------------------- |
| cat_01      | 店内             | オシャレ・雰囲気が良い | 内装、照明、空気感への好意的な評価 |
| cat_02      | 備品・設備       | BGMが大きすぎる        | 音量に関する不満。選曲センスは別   |
| cat_03      | 接客・サービス   | 提供速度が遅い         | 注文から提供までの時間に関する不満 |

### B. 修正データペアテーブル (Recent Corrections)

人間がフィードバック（修正）を行った**直近の具体事例**。AIに「最新のトレンド・言い回し」を教える役割。

- ソースにある「事例をデータ記録して貯める」ためのテーブル。

| id  | raw_segment (入力文節)   | correct_category_id | sentiment | reviewer_comment         | created_at |
| --- | ------------------------ | ------------------- | --------- | ------------------------ | ---------- |
| 101 | 抹茶チーズケーキ絶品です | cat_99 (料理)       | 好意的    | 新メニューのため正解     | 2025-06-08 |
| 102 | 音が悲しげ+拍手が遅い    | cat_02 (設備)       | 不満      | 選曲だが設備（音響）扱い | 2025-04-17 |

### C. ルール一覧テーブル (Compressed Rules)

修正データペアから定期的に抽出・圧縮された**普遍的な判断ルール**。AIに「組織の憲法」を教える役割。

- ソースにある「情報を圧縮して渡す」を実現するテーブル。

| rule_id | target_category | rule_description                                                                                         | priority |
| ------- | --------------- | -------------------------------------------------------------------------------------------------------- | -------- |
| rule_01 | 備品・設備      | 「音響」「BGM」「空調」「Wi-Fi」に関する記述は、「店内（雰囲気）」ではなく「備品・設備」に分類すること。 | High     |
| rule_02 | 接客・サービス  | スタッフの態度だけでなく、「提供スピード」や「子供用カトラリーの提供有無」も接客・サービスに含めること。 | High     |

## 4. 分析アルゴリズムと処理フロー

生成AIエージェントは以下のロジックを実装してください。

### Phase 1: コンテキスト取得 (Retrieval)

ステートレスなAIに対し、判断に必要な情報をDBから取得します。

1. **カテゴリマスター取得:** `Category Master`から全件取得（またはVector検索で関連カテゴリを取得）。
2. **直近事例の取得:** `Recent Corrections`から最新N件（例: 50件）を取得。
3. **ルールの取得:** `Compressed Rules`から有効な全ルールを取得。

### Phase 2: 動的プロンプト構築 (Prompt Engineering)

取得した情報をSystem Promptに埋め込みます。

> System Prompt 構成:
>
> 1. **役割定義:** あなたは顧客の声分析の専門家です。
> 2. **ルール（憲法）:** Table Cの内容を提示。「以下のルールを厳守してください...」
> 3. **Few-Shot事例（直近の傾向）:** Table Bの内容を提示。「以下は直近の分析正解例です...」
> 4. **出力定義:** 後述する3階層JSONモデルに従うこと。
> 5. **カテゴリ制約:** Table Aにあるカテゴリを優先使用すること。該当がない場合のみ `is_new_generated: true` とする。

### Phase 3: 推論と構造化 (Inference)

LLM（OpenAI/Gemini/Claude等）にプロンプトと`raw_text`を渡し、JSONを生成させます。

### アウトプットデータ仕様（3階層モデル）

ソースおよび会話履歴に基づき、**Raw > Segment(文節) > Extract(要素)** の階層構造で出力します。

```
{
  "voice_id": "unique_transaction_id_001",
  "process_timestamp": "2025-05-15T10:00:05Z",
  "segments": [
    {
      "segment_id": 1,
      "input_text": "ランチとは雰囲気が違うしよかったんですが、とにかく音楽がでかい！",
      "extracts": [
        {
          "text_fragment": "ランチとは雰囲気が違うしよかったんですが",
          "sentiment": "好意的",
          "category_id": "cat_01",
          "general_category": "店内",
          "specific_category": "オシャレ・雰囲気が良い",
          "is_new_generated": false
        },
        {
          "text_fragment": "とにかく音楽がでかい！",
          "sentiment": "不満",
          "category_id": "cat_02",
          "general_category": "備品・設備", // ルールにより「店内」ではなく「設備」と判定
          "specific_category": "BGMが大きすぎる",
          "is_new_generated": false
        }
      ]
    },
    {

      "input_text": "あと、ハンバーグが昼とあんまり見栄えに変化がなくて…その分値段の差がちょっとキツいなと感じました。",
```

## 5. 修正フィードバックと精度向上ループ（Education Cycle）

AIの精度（現状60点）を100点に近づけるためのデータの流れです。

### Step 1: 人間による修正 (Human Correction)

クライアントUI（スプレッドシートや管理画面）で、AIの出力結果を確認します。

- 誤りがある場合（例：AIが「BGM」を「店内」としたが、人間が「備品・設備」に直したい）、修正を行います。

### Step 2: 修正データの蓄積 (Feedback API)

修正内容はAPIを通じて即座に `B. 修正データペアテーブル` に保存されます。

- 保存内容: `{ "input": "とにかく音楽がでかい！", "correct": "備品・設備 > BGMが大きすぎる" }`
- **効果:** 次回以降のリクエストで、このデータが「直近事例」としてプロンプトに含まれるため、即座に同じミスをしなくなります（短期記憶の補強）。

### Step 3: ルールの圧縮・生成 (Rule Compression Batch)

**週次バッチ処理**として、以下のプロセスを自動実行します（ソースの概念）。

1. **分析:** 過去1週間に追加された `B. 修正データペア` をLLMに読み込ませる。
2. **抽象化:** 「どのような間違いが多く、どう修正されたか？」を分析させ、共通項を見つけ出す。
   - _事例:_ 「音がうるさい」「BGMが大きい」「スピーカーが割れている」→ 全て「備品・設備」に修正されている。
3. **ルール化:** 新しいルールを生成し、`C. ルール一覧テーブル` に追記・更新する。
   - _新ルール:_ 「音に関する言及は全て『備品・設備』カテゴリに分類せよ」
4. **アーカイブ:** ルール化済みの古い修正データペアはアーカイブし、プロンプトに渡すリストからは除外する（トークン節約）。

## 6. 実装に向けたまとめ

生成AIエージェントへの指示まとめ：

1. **APIエンドポイントを作成せよ:** 生テキストを受け取り、上記JSONを返すPOSTメソッド。
2. **DB接続を実装せよ:** リクエスト毎に「カテゴリマスタ」「最新100件の修正事例」「全ルール」をFetchする処理。
3. **プロンプトテンプレートを作成せよ:** Fetchした情報をSystem Promptに動的に挿入するテンプレート。
4. **構造化ロジックを実装せよ:** 長文を文節（Segments）に区切り、各文節に対して抽出（Extracts）を行う処理。
5. **学習サイクルを構築せよ:** ユーザーの修正を受け取るエンドポイントと、そこからルールを生成するバッチ処理（または定期実行スクリプト）。

この設計により、最初は「一般的なAI」として振る舞いますが、運用を重ねるごとに「その組織特有の判断基準（レシピ）」を持った専属のアナリストへと進化します。
